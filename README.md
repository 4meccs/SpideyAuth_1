# SpideyAuth v3.4 Compatible Backend

Go + Vercel + Supabase backend, **100% protocol-compatible** with `spideyauth_client.lua`.

---

## ⚠️ Critical Corrections vs the Published Spec

The spec document contains several bugs that would break authentication entirely. This implementation fixes all of them based on direct analysis of the Lua client source:

### 1 — PRNG Return Formula (BREAKING)

| | Formula | Example: `prng(100000, 1000000)` |
|---|---|---|
| **Spec (wrong)** | `min + newState % (max-min+1)` | range 100000–1000000 |
| **Actual Lua** | `newState % rangeMax + 1` | range 1–1000000 |

The Lua client code (line 736) is:
```lua
return rangeMin + newState % rangeMax - rangeMin + 1
```
Which simplifies to `newState % rangeMax + 1`. The `rangeMin` parameter **cancels out completely** and is ignored. This implementation uses the correct formula.

### 2 — Cipher Key Indexing (0-based)

The Lua client uses `{ [0] = 0 }` for the initial key — a 0-indexed table. The spec incorrectly describes this as a 1-indexed array. All key positions in this implementation are 0-based.

### 3 — Init Payload Has 13 Fields (not 12)

The spec documents 12 fields. The actual payload has **13 fields** because the client prepends a checksum field (field[0]) before all data fields.

### 4 — Heartbeat Has No Checksum Prepend

Unlike `/init` and `/start`, the heartbeat payload does **not** have a checksum prepend. It is exactly 3 cipher-encoded strings.

### 5 — Start Server Proof Uses JobId Fallback "?"

The client tries the real Roblox `game.JobId` first, then falls back to `"?"`. Since the server cannot know the JobId, this implementation always builds the proof using `stringChecksum("?") = 63`. The client succeeds on its second attempt.

### 6 — Extended Cipher Key Layout

The 8-byte extended key is **interleaved**, not sequential:
```
[ct1%256, extKey1, ct2%256, extKey3, ct3%256, extKey5, ct4%256, extKey7]
```
Positions [0,2,4,6] = client token bytes, [1,3,5,7] = server-generated bytes.

## Setup

### 1. Supabase

1. Create a project at [supabase.com](https://supabase.com)
2. Go to **SQL Editor** and run the entire contents of `schema.sql`
3. Note your **Project URL**, **anon key**, and **service_role key**

### 2. Vercel

1. Install the Vercel CLI: `npm i -g vercel`
2. Clone/push this repo to GitHub
3. Import the repo in [vercel.com](https://vercel.com) → New Project
4. Set these **Environment Variables** in the Vercel dashboard:

| Variable | Value |
|---|---|
| `SUPABASE_URL` | `https://xxxx.supabase.co` |
| `SUPABASE_SERVICE_ROLE_KEY` | your service role key |
| `ENCRYPTION_SECRET` | 32 random bytes (base64) |

5. Deploy: `vercel --prod`

### 3. Configure the Lua Client

In `spideyauth_client.lua`, the host is set via server selection logic. For local testing, add at the top:
```lua
LT_R_RRT_H = "https://your-deployment.vercel.app"
LT_R_RRT_W = "wss://your-deployment.vercel.app/wshttpemu"
```

The client checks `_tremove` (which is set to `ff97f23b97f93792992999 and ff97f23b97f93792992999() == "j" or false`) — for production this is `false`, so it uses the timezone-based server selection. You would instead need to modify the Host value directly.

> **The client must not otherwise be modified** per the spec.

### 4. Add a Script and License Key

Using the Supabase Table Editor or REST API:

```sql
-- Insert a test script
INSERT INTO scripts (id, name, protected_payload, script_note)
VALUES (
  '2674ffd1c9054918b647386bab0ae9fc',  -- matches client ScriptID
  'TEST',
  'print("Hello from SpideyAuth!")',
  'Authentication successful!'
);

-- Insert a test license key
INSERT INTO whitelist_entries (script_id, license_key)
VALUES (
  '2674ffd1c9054918b647386bab0ae9fc',
  'YOUR-LICENSE-KEY-HERE'
);
```

## Protocol Flow (Corrected)

```
Client                                    Server
  │                                         │
  ├─── GET /status ────────────────────────►│
  │◄── { versions: { "3.4": "v1" } } ──────┤
  │                                         │
  │  [Connect WebSocket to /wshttpemu]      │
  │                                         │
  ├─── GET /v1/auth/{scriptId}/init?t=...──►│  ← encrypted with key={0}
  │                                         │    13 fields: checksum + data
  │◄── [16-field response] ─────────────────┤  ← encrypted with 4-byte clientTokens key
  │                                         │
  ├─── GET /v1/auth/start/{urlToken}?t=────►│  ← encrypted with 8-byte extended key
  │                                         │    6 fields: checksum + data
  │◄── [9-field response] ──────────────────┤  ← includes protectedScript
  │                                         │
  │  [Execute protectedScript]              │
  │                                         │
  ├─── GET /v1/auth/heartbeat?t=&s= ───────►│  (every 20s, NO checksum prepend)
  │◄── [1-field response: hash value] ──────┤
  │                                         │
```

### Init Response Field Map (0-indexed)

| Index | Content | Client reads as |
|---|---|---|
| 0 | `maxUses + nonce2` | `initResponse[1] - nonce2 = maxUses` |
| 1 | `extKey5` (byte string) | extended key byte [5] |
| 2 | `transformB + sn2` | `initResponse[3] - sn2 = authModifier` |
| 3 | `expiryTimestamp + nonce1` | `initResponse[4] - nonce1 = expiryTimestamp` |
| 4 | `transformC + sn3` | `initResponse[5] - sn3 = authTransformer` |
| 5 | `extKey7` (byte string) | extended key byte [7] |
| 6 | `extKey3` (byte string) | extended key byte [3] |
| 7 | `transformA + sn1` | `initResponse[8] - sn1 = authPayload` |
| 8 | `extKey1` (byte string) | extended key byte [1] |
| 9 | `sessionToken` (numeric) | used in heartbeat hash |
| 10 | `serverProof` (hash concat) | identity verification |
| 11 | `sessionURLToken` | URL path + `?s=` param |
| 12 | `startHashInput` | included in start payload |
| 13 | `maxUsesLimit` or `"-1"` | usage cap |
| 14 | `totalUses` | current use count |
| 15 | `discordId` | shown to user |

## WebSocket Note

Vercel Serverless Functions have a **10-second max duration** and don't support persistent connections. The WebSocket handler at `/wshttpemu` **cannot run on Vercel**. Options:

- **Railway / Fly.io**: Deploy the websocket handler as a separate long-running service pointing to `wss://your-ws-host/wshttpemu`
- **Vercel Edge Functions**: Experimental WebSocket support via `@vercel/edge`
- **Fallback**: The client automatically falls back to HTTP if WebSocket fails — the auth flow still works

## Adding Discord Bot

The Discord slash commands (`/whitelist add`, `/whitelist remove`, `/resethwid`, `/userinfo`) are not included in the Vercel functions since Discord's Interactions webhook model requires a separate endpoint. Recommended: add `api/discord.go` that handles `POST /api/discord` with signature verification using `DISCORD_PUBLIC_KEY`, then dispatches commands to the Supabase DB.
